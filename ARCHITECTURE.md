## 📊 Обзор системы

Система отслеживания грузов (Shipments Tracking Service) — микросервисная архитектура для управления доставкой с уникальными трекинг-номерами.

**Основной функционал:**
- Создание груза с уникальным трекинг-номером
- Обновление статуса груза (pending → in_transit → delivered)
- Поиск груза по ID или трекинг-номеру
- Удаление груза
- Асинхронные уведомления об изменении статуса

### 1. Gateway (Nginx)

| Параметр     | Значение                                   |
|--------------|--------------------------------------------|

| Порт         | 8085 (внешний) → 8130 (внутренний)         |
| Назначение   | Reverse proxy, маршрутизация, балансировка |

**Функции:**

### 2. shipments-svc-s01 (Основной сервис)

| Параметр     | Значение                                     |
|--------------|----------------------------------------------|
| Образ        | shipments-s01:latest                         |
| Порт         | 8130 (REST)                                  |
| Фреймворк    | FastAPI + Uvicorn                            |
| Язык         | Python 3.13                                  |
| Хранилище    | In-Memory (список)                           |
| Протокол     | REST/JSON (внешний), gRPC-style (внутренний) |

**API Endpoints:**

| Метод  | Путь                                 | Описание                 | Статус  |
|--------|--------------------------------------|--------------------------|---------|
| GET    | `/`                                  | Health check сервиса     | 200     |
| GET    | `/health`                            | Проверка готовности      | 200     |
| POST   | `/api/shipments`                     | Создать груз             | 201     |
| GET    | `/api/shipments`                     | Получить все грузы       | 200     |
| GET    | `/api/shipments/{id}`                | Получить груз по ID      | 200/404 |
| GET    | `/api/shipments/tracking/{tracking}` | Поиск по трекинг-номеру  | 200/404 |
| PUT    | `/api/shipments/{id}`                | Обновить статус груза    | 200/404 |
| DELETE | `/api/shipments/{id}`                | Удалить груз             | 204/404 |

**Модель данных (Shipment):**

class Shipment(BaseModel):
    id: int                    # Уникальный ID
    tracking: str              # Трекинг-номер (уникальный)
    origin: str                # Откуда
    destination: str           # Куда
    status: str                # pending/in_transit/delivered
    created_at: float          # Время создания
    updated_at: float          # Время последнего обновления


**Логика уведомлений:**
При создании или обновлении груза сервис асинхронно отправляет событие:

📧 [gRPC → notifications-svc] Shipment TRK-XXX → new_status

## 🔄 Потоки данных

### Создание груза (Create Shipment)

Client → Gateway:8085 (POST /api/shipments)
    → shipments-svc-s01:8130 (валидация через Pydantic)
    → In-Memory Storage (добавление в список)
    → [Async] notifications-svc-s01:8131 (событие: "created")
    → Client (201 Created + JSON)


### Обновление статуса (Update Status)

Client → Gateway:8085 (PUT /api/shipments/{id})
    → shipments-svc-s01:8130 (поиск по ID)
    → In-Memory Storage (обновление status + updated_at)
    → [Async] notifications-svc-s01:8131 (событие: "status_changed")
    → Client (200 OK + Updated JSON)


### Поиск по трекингу (Track Shipment)

Client → Gateway:8085 (GET /api/shipments/tracking/{tracking})
    → shipments-svc-s01:8130 (линейный поиск по списку)
    → Client (200 OK + Shipment JSON или 404)

### 🔴 IDOR — Критическая уязвимость

**Описание:** Эндпоинт `PUT /api/shipments/{id}` позволяет изменить данные любого груза, зная его ID. Проверка владельца отсутствует.

**Как воспроизвели:**
1. Создан груз жертвы (ID=3): `{"tracking":"VICTIM-001","origin":"Moscow",...}`
2. Отправлен запрос атакующего: `PUT /api/shipments/3` с данными `{"origin":"STOLEN_DATA"}`
3. Сервер вернул `200 OK` и обновил груз жертвы

**Рекомендация:** Внедрить JWT-авторизацию и проверять `token.user_id == shipment.owner_id`.

## ⚠️ Ограничения и улучшения

### Текущие ограничения
1. In-Memory хранилище — данные теряются при перезапуске
2. Нет JWT-авторизации — любой может изменить любой груз (IDOR)
3. Нет реальной отправки уведомлений — только логирование
4. Нет Rate Limiting — уязвимость к спаму/DoS
